## Постановка Задачи

Проект решает небольшую задачу точного program synthesis:

> Дана функция-черный ящик `f: uint8 -> uint8`. Нужно найти компактное выражение `e(x)`, такое что `e(x) == f(x)` для каждого `x` от `0` до `255`.

Область определения намеренно конечная. Вместо доказательства эквивалентности на бесконечном множестве программа может вычислить целевую функцию и каждого кандидата на всех 256 возможных входах. Кандидат считается корректным только тогда, когда его полная 256-байтная таблица значений совпадает с таблицей целевой функции.

На выходе можно получить:

- выражение в человекочитаемом виде, например `((x >> 2) + (x ^ 17))`;
- отчет о найденном и проверенном результате;
- Verilog-модуль, реализующий выражение как комбинационную логику.

## Язык Выражений

Синтезатор ищет выражения в фиксированном DSL для 8-битной арифметики. Все операции выполняются по модулю 256.

Поддерживаемые виды выражений описаны в `cpp_port/include/synth/core.hpp`:

- `x`;
- константы;
- умножение на константу;
- сложение и вычитание;
- побитовый XOR;
- левый и правый сдвиг на константу;
- квадрат;
- общее умножение;
- периодические табличные операции `Sin8` и `Cos8`.

Каждое выражение хранит:

- тип операции и дочерние узлы;
- стабильный id выражения;
- стоимость;
- текстовый ключ формы выражения;
- метаданные о литеральных константах и глубине нелинейности.

Стоимость используется как основной порядок поиска: более дешевые выражения рассматриваются раньше, а кандидат с меньшей стоимостью предпочтительнее более дорогого эквивалентного кандидата.

## Построение Цели

Цель представляется как `Semantics256`: массив из 256 байт. Он строится вычислением oracle для каждого возможного байтового входа:

```text
target[x] = oracle(x) & 0xFF, для x в 0..255
```

В обычном запуске oracle - это `current_oracle` из `cpp_port/src/current_oracle.cpp`. В benchmark-режиме oracle берется из `cpp_port/src/benchmarks.cpp`.

Дополнительно система вычисляет более короткую `Signature24`: значения цели на 24 выбранных точках. Эта сигнатура используется как дешевое приближение во время поиска, до полной оценки кандидата.

## Bottom-Up Enumerative Search

Основной поиск - это перечисление выражений снизу вверх по стоимости.

На стоимости 1 поиск начинается с:

- `x`;
- базового набора констант.

Для каждого следующего уровня стоимости строятся новые кандидаты из выражений, найденных на более низких уровнях. Унарные операции применяются к выражениям предыдущего уровня, а бинарные операции комбинируют пары более дешевых выражений так, чтобы суммарная стоимость соответствовала текущему уровню.

Основной цикл находится в `find_enumerative_round` в `cpp_port/src/core.cpp`.

На верхнем уровне процесс выглядит так:

```text
for cost = 1..max_cost:
    сгенерировать выражения-кандидаты с этой стоимостью
    упростить/канонизировать каждое выражение
    вычислить его сигнатуру
    при необходимости отбросить по signature bucket
    вычислить полную 256-байтную семантику
    сохранить, если это лучшее выражение для такой семантики
    остановиться, если семантика совпала с целью
```

Важное свойство: поиск сравнивает не только синтаксис, а поведение. Два разных выражения, которые вычисляют одну и ту же 256-байтную таблицу, считаются семантически эквивалентными, и из них сохраняется лучшее.

## Канонизация

Smart constructors упрощают многие выражения до попадания в таблицы поиска.

Примеры:

- `x + 0` превращается в `x`;
- `x - x` превращается в `0`;
- `x ^ x` превращается в `0 (^ - XOR, операция возведения в степень нативно не реализована)`;
- `(x ^ 17) ^ 5` превращается в `x ^ 20`;
- `(3 * x) * 5` превращается в `15 * x`;
- `x * x` превращается в `Square(x)`;
- выражения только из констант сразу сворачиваются в константу.

Это уменьшает пространство поиска и помогает тестам сравнивать стабильные ожидаемые выражения.

## Interner И Семантическая Дедупликация

Выражения хранятся в арене и передаются как `const Expr*`. Interner гарантирует, что структурно одинаковые узлы переиспользуются.

Поиск также хранит отображение от полной семантики к лучшему известному кандидату. Если новое выражение вычисляет таблицу, которая уже встречалась, оно отбрасывается, если только не лучше по ранжированию проекта:

1. меньшая стоимость;
2. более короткий shape key;
3. лексикографически меньший shape key.

Благодаря этому поиск может обрабатывать много эквивалентных выражений, не сохраняя их все.

## Signature Pruning

Полная оценка на 256 входах дешевая по сравнению со многими задачами синтеза, но количество кандидатов все равно быстро растет. Поэтому проект использует 24-точечную сигнатуру как ранний фильтр.

Для каждого кандидата поиск вычисляет:

- `Signature24`: значения на выбранных точках;
- `Semantics256`: значения на всех точках, только если кандидат пережил раннее отсечение.

Для каждой сигнатуры поиск хранит небольшой bucket хороших выражений. Если новое выражение попадает в уже заполненный bucket и хуже текущих выражений в нем, оно может быть отброшено до полной оценки.

Доступны две стратегии:

- `top1` хранит одно лучшее выражение на bucket сигнатуры;
- `selective` разрешает более крупные buckets на больших стоимостях.

Чтобы сохранить точность, основной поиск имеет fallback: если проход с signature pruning не нашел результат, поиск повторяется без сигнатурного отсечения.

## Promotion Констант

Базовый набор констант намеренно ограничен. Во время поиска синтезатор может обнаруживать полезные константы, которые появляются внутри упрощенных выражений. Такие константы добавляются в доступный набор для следующего раунда.

Поэтому некоторые выражения, которые неудобно найти только с начальными константами, становятся достижимыми после одного или нескольких promotion-раундов.

Информация о таких раундах записывается в `PromotionRound`.

## Режимы Нелинейности

Нелинейные выражения сильно увеличивают пространство поиска. В проекте есть три режима:

- `restricted`: разрешает только неглубокое использование нелинейности; режим по умолчанию;
- `relaxed`: разрешает больше нелинейных композиций;
- `open`: самый широкий режим нелинейного поиска из текущих.

Режим управляет:

- максимальным nonlinear count;
- тем, разрешено ли умножение с нелинейными операндами.

Это практический компромисс. Более свободные режимы позволяют выразить больше функций, но могут заметно замедлить поиск.

## CEGIS-Режим

В проекте также есть sample-based режим в стиле CEGIS.

Вместо немедленного сравнения кандидатов на всех 256 входах он начинает с меньшей выборки. Если кандидат совпадает с выборкой, программа проверяет его на полном домене. Если проверка не проходит, вход с ошибкой добавляется как контрпример, и поиск повторяется.

Идея взята из Counterexample-Guided Inductive Synthesis: синтезатор строит кандидата по текущему набору примеров, verifier ищет контрпример, а найденный контрпример добавляется в набор примеров для следующего раунда.

Схема:

```text
sample = начальные выбранные точки
repeat:
    синтезировать выражение, совпадающее с sample
    проверить на всех 256 входах
    if verified: успех
    else добавить counterexample в sample
```

Этот режим полезен для изучения oracle-guided synthesis и сбора метрик, хотя для 8-битного проекта full-domain режим проще и надежнее.

## Верификация

Верификация точная, потому что пространство входов содержит только 256 значений.

Для выражения-кандидата `expr` функция `verify_expression` вычисляет:

```text
expr(0), expr(1), ..., expr(255)
```

и сравнивает результат с целевой таблицей. Если какой-то вход отличается, верификатор возвращает первый ошибочный вход. Иначе выражение принимается как полностью эквивалентное oracle на 8-битном домене.

## Генерация Verilog

После того как выражение найдено и проверено, `emit_verilog` записывает модуль:

```verilog
module generated_function (
    input  wire [7:0] x,
    output wire [7:0] y
);
    assign y = ...;
endmodule
```

Если выражение использует `Sin8` или `Cos8`, генератор добавляет небольшие Verilog-функции с `case`-таблицами на 256 значений. Затем сгенерированный модуль подключается в `fpga_demo/top.sv` к switches, LEDs и seven-segment displays.

## Benchmarks

Benchmark-кейсы описаны в `cpp_port/src/benchmarks.cpp`. Они покрывают:

- линейные функции;
- смеси сдвигов и XOR;
- случаи для promotion констант;
- умножение на константу;
- периодические случаи `Sin8`/`Cos8`;
- нелинейные случаи с квадратом и умножением;
- отрицательные случаи, которые не должны находиться при стандартных ограничениях.

Benchmark runner сообщает, была ли найдена формула, прошла ли она проверку, стоимость выражения, время backend, pruning-метрики и при необходимости пишет JSON.

## Основные Ограничения

Текущая реализация намеренно узкая:

- поддерживается один 8-битный вход и один 8-битный выход;
- все поведение считается по модулю 256;
- пространство поиска ограничено `max_cost`;
- *DSL* зафиксирован в C++;
- функции вне DSL или выше заданного лимита стоимости не будут найдены;
- более широкие нелинейные режимы могут быть намного медленнее;
- сгенерированный Verilog комбинационный и рассчитан на простой demo-use;
- `exact-byte-synth.qsf` все еще требует реальных настроек платы, устройства и pin assignments перед использованием в Quartus если использвать любую другую плату;
- поиск оптимизирован под конечный 8-битный случай, а не под произвольную ширину слова или много входов.

Для учебного проекта эти ограничения приемлемы: они делают точную full-domain проверку простой и сохраняют FPGA-demo понятным.

## References

Проект вдохновлен program synthesis и component-based bit-vector synthesis, но устроен проще: он использует небольшой вручную заданный DSL и точную проверку на 256 входах вместо общего SMT-based synthesis framework.

Ближайшие референсы:

1. [Synthesis of Loop-Free Programs - Microsoft Research](https://www.microsoft.com/en-us/research/publication/synthesis-loop-free-programs/)
  Работа описывает синтез loop-free программ из библиотек компонентов. Она близка к этому проекту, потому что обе системы ищут небольшие программы над битовыми и арифметическими компонентами.
2. [Oracle-Guided Component-Based Program Synthesis - Microsoft Research](https://www.microsoft.com/en-us/research/publication/oracle-guided-component-based-program-synthesis/)
  Это самый близкий концептуальный референс для oracle-guided постановки: синтезатор строит программу, взаимодействуя с источником input/output поведения.
3. [fitzgen/synth-loop-free-prog](https://github.com/fitzgen/synth-loop-free-prog)
  Rust/Z3-реализация, вдохновленная направлением loop-free synthesis. Это полезная практическая точка сравнения: похожие идеи реализованы через SMT solver, а не через прямой enumerative search.
4. [Program Synthesis By Sketching - EECS at UC Berkeley](https://www2.eecs.berkeley.edu/Pubs/TechRpts/2008/EECS-2008-176.html)
  Это один из канонических источников для CEGIS/SKETCH-подхода. Из этой линии работ взята идея итерации `synthesize -> verify -> add counterexample`, которая в проекте используется в sample-based режиме.

От этих работ проект отличается очень маленьким конечным доменом и тем, что частью workflow является FPGA-ready Verilog artifact.
